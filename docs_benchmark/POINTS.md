# Research Paper Points — docs_benchmark

Source-of-truth bullet list of every finding worth using in the paper. Organized by where they belong in the write-up. Numbers in parentheses are bootstrap CIs at 95%, paired bootstrap over n=143 unless noted. Model: gemini-3-flash. Dataset: coverage set (120 standard + 23 forced-inconsistency = 143 records).

---

## 1. Benchmark design (Section: Dataset & Methodology)

- **Synthetic accounting-document reconciliation task.** Each record is a packet of OCR'd documents (invoices, statements, payment notices, schedules, certificates, distractors). The model must produce (a) journal entries with `doc_refs`, (b) the final balance sheet, and (c) inconsistency flags + codes for unreconcilable packets.
- **Deterministic ledger ground truth.** Every record is generated by a double-entry ledger replay; GT entries balance, GT balance sheet ties, opening trial balance balances. No human-labeled examples — every label is computable.
- **Eight industries × three period types (month/quarter/year) × five difficulty levels.** Difficulty is defined by: number of documents, cross-document dependencies, jurisdictional/tax complexity, temporal lookback depth, and concept presence (ASC 606, leases, deferred tax, asset disposals, FX, exemptions).
- **Concept coverage flags per record.** `has_asc606`, `has_asset_disposal`, `has_deferred_tax`, `has_lease`, `has_tax_exemption`, etc. — enables per-concept slice analysis.
- **Forced inconsistency packets.** 23 records (16%) contain a deterministic contradiction belonging to one of 24 codes (e.g., `bank_closing_mismatch`, `invoice_total_mismatch`, `tax_total_mismatch`). Models must detect AND classify.
- **Provider-neutral ablation harness.** Orthogonal three-axis matrix: 4 prompt variants × 10 visibility variants × 7 tool variants. Each ablation is an `AblationSpec` triple, run through one shared evaluation loop. Outputs JSONL + summary.json + slice_tables/ per ablation.
- **Bootstrap analyzer.** Paired bootstrap (n=5000 iterations, shared sample indices across baseline and ablation) over 6 metrics. Reports diff + CI per (ablation, metric) pair.
- **Cost.** Coverage set runs at $0.006/record baseline (~$0.90 for all 143). Forced-verifier 1-pass is $0.013/record. Total ablation suite is ~$15–20 per model sweep.

## 2. Headline findings (Section: Results — Lead with these)

### 2.1 Aggregation gap — central finding

- **BS_recon (oracle replay of model's entries) vs BS_exact (model's self-reported BS) gap: 37.5pp on baseline** (62.5% vs 25.0%).
- The gap grows monotonically with difficulty: L1=12.5pp, L2=37.5pp, L3=50.0pp, L4=42pp, L5=46pp.
- The model can write correct entries but does not aggregate them correctly into its own reported BS. The capability is present; the self-consistency is not.

### 2.2 Forced ledger verifier closes the aggregation gap completely

- **+32.5pp BS_exact, CI [+24.0, +40.8]** for 1-pass verifier; **+35.8pp [+26.9, +44.9]** for 2-pass. Both highly significant.
- After verifier feedback: BS_recon ≈ BS_exact at every difficulty level (gap < 4pp). 2-pass closes gap to 0pp.
- The model converges its self-reported BS to its own replayed BS when given deterministic feedback showing the deltas.
- **Mechanism is self-correction of aggregation, not of entries.** Entry-level metrics (BS_recon, JE_strict, account/amount accuracy) are *unchanged* by the verifier. Error taxonomy shows arithmetic and account-selection errors basically constant; only the surface BS_exact metric moves.
- **Interpretation:** the aggregation gap is a self-consistency failure, not a capability failure. This is the paper's reframing of the headline finding.

### 2.3 Doc-linking is the dominant entry-level failure mode

- **Baseline: 52% of expected entries (978/1873) have correct account + amount + date but WRONG doc_refs.**
- This single failure mode is why strict joint BS+JE metric collapses to 5.0% despite 56.7% entry-accounting record-match.
- `entries_accounting_correct_but_doc_refs_wrong_rate` = 0.50 in baseline.
- Decouple strict-doc-refs and lenient-doc-refs as two separate metrics in the paper. The 5% strict number is not "the model fails" — it's "the model can't cite."

### 2.4 Evidence-only context (oracle retrieval) confirms doc-linking is a retrieval problem

- Removing all non-evidence documents reduces strict_doc_ref_mismatch_rate by **−26.4pp [−29.8, −23.0]**.
- BS+JE joint match improves **+5.0pp [+1.6, +9.3]**.
- BS_recon *decreases* −10.0pp [−17.1, −3.4] — removing support docs loses some private reasoning material.
- **Net interpretation:** doc-linking failure is a disambiguation problem (too many candidate docs), not a content problem (the model knows what to cite when shown only correct candidates).

### 2.5 Distractor stress saturates fast

| Variant | BS_recon | Δ vs baseline | inc_code |
|---|---|---|---|
| baseline | 0.625 | — | 0.870 |
| +5 distractors | 0.458 | −16.7pp | 0.652 |
| +15 distractors | 0.467 | −15.8pp | 0.565 |
| +30 distractors | 0.442 | −18.3pp | 0.652 |
| relevant_last | 0.492 | −13.3pp | **0.870** |

- One-time penalty (~−15pp on BS_recon) for any distractors. Adding more from 5 → 30 only loses an additional 1.6pp.
- **Position bias on inconsistency detection:** putting evidence at the END of the document list preserves inconsistency code-match at baseline (0.870), while putting it first drops it to 0.55–0.65.
- Position bias does NOT recover BS reasoning — `evidence_relevant_last` BS_recon is still −13.3pp.

## 3. Negative results (Section: Results — Strong negative findings)

### 3.1 Models bypass offered tools

- Across `tool_calculator`, `tool_document_search`, `tool_ledger_check`, `tool_full_tool_agent` (143 records each, 4 variants): **`tool_call_count_total = 0`** across the board.
- The model emits the final benchmark JSON directly on turn 1; the agent loop short-circuits via `_looks_like_submission`.
- Lightly-offered tools without forcing don't help. Frame as: prompt-time tool availability ≠ tool use.
- This is *why* the `forced_ledger_verifier` variant was added — to test what happens when the model is *required* to receive deterministic feedback.

### 3.2 Chain-of-thought variants don't help and hurt inconsistency detection

- `prompt_guided_private_solve` (structured 6-step workflow): BS_recon 0.625 vs 0.625 baseline. **inc_code_match: 0.739 vs 0.870 (−13pp).**
- `prompt_self_check` (workflow + verification): BS_recon 0.567 vs 0.625. **inc_code_match: 0.783 vs 0.870 (−9pp).**
- CoT on this domain mildly *hurts* inconsistency detection — model reasons around contradictions and over-rationalizes a posting.

### 3.3 Prompt-level "compute BS from entries" instruction does not close the aggregation gap

- `prompt_balance_reconstruction` (explicit instruction to derive BS only by replaying entries): BS_recon=0.600, BS_exact=0.275 (n=40 targeted) — essentially baseline (0.625/0.275).
- **The model cannot be talked into self-consistency via prompting alone.** Only the programmatic verifier achieves it.
- This *strengthens* the forced-verifier finding: it's not something a careful prompt can replicate.

### 3.4 Second verifier pass is wasted cost

- 2-pass vs 1-pass: BS_exact +3.3pp, BS_recon +1.6pp. Cost +56%.
- 2-pass inc_code_match: **−21.7pp [−40.0, −5.3]** significant degradation.
- 1-pass is the practical sweet spot. Recommend dropping 2-pass from headline tables.

## 4. Slice analysis (Section: Results — Per-slice breakdowns)

### 4.1 By difficulty (baseline → forced 1-pass; BS_recon → BS_exact gap)

| Level | baseline gap | forced gap |
|---|---|---|
| L1 | 12.5pp | 0pp |
| L2 | 37.5pp | 0pp |
| L3 | 50.0pp | 4pp |
| L4 | 42pp | 4pp |
| L5 | 46pp | 0pp |

### 4.2 By industry (BS_recon baseline / forced 1-pass)

| Industry | baseline BS_recon | forced BS_recon | baseline BS_exact | forced BS_exact |
|---|---|---|---|---|
| healthcare_clinic | 1.000 | 1.000 | 0.200 | 1.000 |
| property_management | 0.933 | 0.800 | 0.467 | 0.800 |
| field_services | 0.667 | 0.600 | 0.200 | 0.600 |
| wholesale_distribution | 0.667 | 0.600 | 0.267 | 0.600 |
| manufacturing | 0.533 | 0.467 | 0.267 | 0.467 |
| professional_services | 0.400 | 0.467 | 0.133 | 0.467 |
| retail | 0.400 | 0.400 | 0.200 | 0.267 |
| subscription_saas | 0.400 | 0.400 | 0.267 | 0.400 |

- Hard industries (retail 40%, SaaS 40%, pro_services 40%) are limited by entry quality. The verifier can't rescue entries it doesn't have.
- Easy industries (healthcare, property_mgmt) have high BS_recon but low BS_exact — the verifier gives them ceiling performance.

### 4.3 Visibility-collapse industries (`support_docs_removed`, `no_allowed_accounts`)

- `support_docs_removed`: BS_recon collapses 0.625 → 0.250 (−37.5pp). SaaS and wholesale go to 0%.
- `no_allowed_accounts`: BS_recon collapses 0.625 → 0.158 (−46.7pp). L3–L5 go to 0%.
- These prove (a) support documents are critical, not decoration; (b) the allowed-accounts list is a strong scaffold the model relies on.

## 5. Error taxonomy (Section: Error analysis)

Baseline error counts on 120 standard records:

| Error type | Affected records | Rate | Entry count |
|---|---|---|---|
| document_linking_failure | 111/120 | 92.5% | 981 entries |
| omission_failure | 51/120 | 42.5% | 235 entries |
| balance_reconstruction_failure | 45/120 | 37.5% | — |
| hallucinated_extra_entry | 37/120 | 30.8% | 139 entries |
| arithmetic_failure | 25/120 | 20.8% | 37 entries |
| account_selection_failure | 10/120 | 8.3% | 14 entries |
| inconsistency_code_failure | 3/23 | 13% | — |

- **Doc-linking dwarfs every other error.** ~9× the entry count of the next-worst category.
- Account selection (the most "knowledge-y" failure) is the *least* common entry error.
- Hallucinated extra entries (1.16 per standard record) are nearly as common as missing entries (1.96).

## 6. Self-consistency reframing (Section: Discussion — the key conceptual move)

- The benchmark's central failure mode is *not* "models can't do accounting." It is "models produce locally-correct entries that they do not consistently aggregate into their own balance sheet, and they cite the wrong supporting documents."
- Both of these are **process failures, not knowledge failures**:
  - Aggregation gap closes completely with a programmatic verifier (Sec 2.2).
  - Doc-linking failure halves with oracle retrieval (Sec 2.4).
- A capable accounting model wrapped in a thin programmatic shell (verifier + retrieval) recovers most of the gap. A bare LLM prompted in isolation does not.
- This is the framing the paper should lead with: **synthetic-document reconciliation benchmarks reveal a self-consistency / process gap distinct from a capability gap.**

## 7. Methodology notes (Section: Limitations / Reproducibility)

- **Single-seed runs at temp=0.0.** Bootstrap CIs are on the 143 records, not across seeds. For variance under stochastic decoding, additional seeds are needed. Recommend n=3 seeds for headline conditions before camera-ready.
- **Single-model results.** All numbers above are gemini-3-flash. Multi-model sweep is the highest-priority missing experiment. Even n=40 stratified per model would suffice to claim generality of the aggregation gap.
- **Evidence_only is an oracle condition.** It uses GT `doc_refs` to filter visible documents. Frame in methods section as an upper bound, not a realistic deployment condition.
- **Verifier feedback bias on inconsistency packets** is patched: when `parsed.has_inconsistency=True`, BS-comparison is skipped (returns `matches=None, delta_count=0`). Verified by per-record inspection: only 1/19 (1-pass) and 2/19 (2-pass) inconsistency drafts were flipped to non-inconsistency in the final answer.
- **OpenRouter determinism is not guaranteed even at temp=0.0.** Differences smaller than ~3pp between conditions should be reported as not robust.

## 8. Recommended additional experiments before submission

### Must-do
1. **Multi-model sweep.** gemini-3-pro, Claude Sonnet 4.6, GPT-5, DeepSeek-V3, Llama-3.3-70B, and one small open model (Qwen-2.5-32B). Run at minimum the headline set: baseline, forced_ledger_verifier (1-pass), evidence_only, evidence_plus_15_distractors.
2. **Variance.** n=3 seeds on baseline + forced_ledger_verifier. Report mean ± std on BS_recon, BS_exact, inc_code_match.
3. **Per-concept ablation.** Filter the 143 records by concept flag (lease, ASC 606, deferred tax, asset disposal, tax exemption) and report per-concept BS_recon. Reviewers will ask "which accounting concepts does the model fail at?"

### Should-do
4. **Forced-verifier with entry-level feedback** — current verifier only checks BS aggregation. A variant that also flags `doc_refs` mismatches or amount discrepancies against the deterministic ledger would test whether stronger feedback also fixes entries (not just aggregation).
5. **Self-consistency (k=5 majority vote)** at temp=0.7 on baseline. If BS_recon jumps to 80%+, the model is making lucky-bad guesses; if flat, errors are systematic.
6. **Account-naming robustness.** Replace `Accounts Receivable` → `Trade Debtors`, etc. (UK/IFRS naming). Isolates accounting-conceptual failure from US-GAAP string-matching.

### Nice-to-have
7. **OCR-degradation ablation** (1–3% char noise on OCR text). Real-world deployment story.
8. **Few-shot ablation** (0-shot / 1-shot / 3-shot). Reviewers expect this; cheap to run.
9. **Doc-order shuffling** (already partially covered by `evidence_relevant_last`).
10. **Tool quality ablation** — return a noisy ledger 20% of the time. Tests model's rejection of bad oracle feedback.

## 10. Code-side facts useful for methods section

- Deterministic ledger in `docs_benchmark/ledger.py` enforces double-entry, contra-asset handling for accumulated depreciation, revenue/expense closure into retained earnings.
- Opening trial balances are back-solved to balance: assets + liabilities first, equity residual.
- Document types: `posting_doc`, `support_doc`, `adjustment_doc`, `distractor_doc`, `opening_trial_balance`. Visibility ablations filter on `role`.
- Parser (`docs_benchmark/benchmark/parser.py`) handles fenced JSON, repairs thousands separators, enforces inconsistency codes from `INCONSISTENCY_CODES`.
- Verifier compares submitted vs reconstructed BS at 0.01 tolerance, with section-by-section + total-by-total deltas. Inconsistency-claimed drafts skip the comparison.
- Bootstrap is paired (shared sample indices for baseline and ablation) for matched-record comparisons.
- Stratified sampler in `dataset.py` round-robins across (difficulty_level, industry) groups deterministically.
