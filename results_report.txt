This document presents all evaluation results collected for the FinBalance
benchmark, including headline metrics, dimension-by-dimension failure analysis,
error propagation traces, and an assessment of the findings in the context of
a top-tier AI/NLP conference submission.

PART 1 — EXPERIMENT OVERVIEW
--------------------------------------------------------------------------------

Dataset split  : data/sample.jsonl (25 of 100 problems evaluated)
Difficulty dist: L1=5, L2=5, L3=5, L4=5, L5=5
Model backend  : Anthropic Messages API (claude-3-haiku-20240307)
Temperature    : 0 (near-deterministic)
Prompting      : zero_shot (bare instruction + transaction list → JSON)
Parse failures : 0 / 25  (0%)

NOTE: The parser was significantly hardened during this session after a root
cause analysis revealed that the model outputs monetary values with
thousands-separator commas (e.g. "Cash": 123,900) which json.loads() rejects.
The fix uses a regex substitution to strip internal commas from numbers before
parsing. All 25 responses parsed successfully after this fix.

Metric definitions:
  BA  = 1 if |assets − (liabilities + equity)| < $1, else 0
  ALA = correct_accounts / total_expected_accounts
  TPA = fraction of intermediate account states correctly predicted
        (zero_shot strategy does not output intermediate states → always 0)
  FBS = 0.4 × BA  +  0.4 × ALA  +  0.2 × TPA

Interpretation:
  - The model correctly balances the balance sheet in only 20% of problems.
  - It gets individual account values right ~42% of the time overall.
  - The composite score of 36.3/100 places it well below functional competence
    for automated financial reporting.
  - TPA = 0% is expected under zero_shot (model outputs only the final sheet,
    not step-by-step ledger states). This dimension is reserved for CoT/self-
    refine evaluations.

PART 3 — PERFORMANCE BY DIFFICULTY LEVEL
--------------------------------------------------------------------------------

  Level  N    BA       ALA      FBS    Avg Errors  Avg Accounts Expected
  -----  --   ------   ------   -----  ----------  ---------------------
  L1      5   40.0%    0.5333   45.9      2.8            5.0
  L2      5   60.0%    0.5129   51.5      4.6            8.6
  L3      5    0.0%    0.4817   32.0      7.6           12.2
  L4      5    0.0%    0.3636   29.2     10.4           14.4
  L5      5    0.0%    0.2241   23.2     12.6           15.0

Key observations:

  1. Performance cliff at L3. BA collapses from 60% (L2) to 0% (L3) and never
     recovers. L3 introduces COGS tracking and accrual adjustments — the model
     cannot handle these correctly.

  2. ALA degrades monotonically across levels (0.533 → 0.224), confirming that
     even partial credit decreases as complexity increases.

  3. Average error count scales with problem complexity: 2.8 errors per problem
     at L1, 12.6 errors per problem at L5.

  4. At L5, the model correctly accounts for only 3.4 out of 15 expected
     accounts on average — a 77% miss rate.

  5. L2 outperforms L1 on BA (60% vs 40%). This is a small-sample effect
     (n=5 each); L1 problems in this sample included edge cases where the
     model omitted the liabilities section when there were loans present
     (missing_section failures), while L2 problems happened to have simpler
     liability structures.

PART 4 — ERROR COMPOSITION BY LEVEL
--------------------------------------------------------------------------------

Error type codes:
  AE = Arithmetic Error   (account present, value wrong)
  OE = Omission Error     (expected account not predicted)
  CV = Classification     (account in wrong balance sheet section)
  CO = Count/Order        (duplicate entries)

  Level   AE%     OE%     CV%     CO%    Total Errors
  -----   -----   -----   -----   ----   ------------
  L1      42.9%   35.7%   21.4%    0%       2.8
  L2      69.6%   21.7%    8.7%    0%       4.6
  L3      60.5%   21.1%   15.8%   2.6%      7.6
  L4      73.1%   15.4%   11.5%    0%      10.4
  L5      82.5%    9.5%    7.9%    0%      12.6

Key trend: As difficulty increases, the error composition shifts heavily toward
Arithmetic Errors (AE). At L5, 82.5% of all errors are arithmetic — the model
has the right *accounts* but computes wrong *values*. This indicates that at
lower levels the model makes conceptual errors (omitting whole accounts),
whereas at higher levels the model's schema knowledge is partially there but
its transaction arithmetic breaks down under compounding complexity.

PART 5 — COMPLEXITY FEATURE IMPACT ON PERFORMANCE
--------------------------------------------------------------------------------

The failure analysis correlates problem-level complexity features with FBS
degradation. ΔFBS is the mean FBS difference between problems that do vs do not
contain the given feature (negative = feature hurts performance).

  Feature               N problems   Avg FBS   BA Rate   ΔFBS    ΔALA
  -------------------   ----------   -------   -------   -----   ------
  prepaid               24 / 25      35.39     16.7%    -23.71  +0.094
  cogs                  15 / 25      28.09      0.0%    -20.61  -0.167
  derived_interest      10 / 25      26.14      0.0%    -16.99  -0.188
  mixed_funding         10 / 25      26.16      0.0%    -16.96  -0.216
  debt                  18 / 25      32.72     11.1%    -12.90  -0.081
  credit_transaction    20 / 25      33.94     15.0%    -11.99  -0.138
  depreciable_asset     23 / 25      35.51     17.4%    -10.30  -0.047

Findings:

  1. COGS (ΔFBS = -20.61, BA = 0%): When the model must track inventory
     contra cash/AR AND record COGS separately, it never balances. The 4-entry
     nature of COGS transactions (Dr Cash/AR, Cr Revenue, Dr COGS, Cr Inventory)
     appears to be beyond the model's reliable capability.

  2. Derived interest (ΔFBS = -16.99): Computing interest as loan_balance × rate
     / 12 and accruing it correctly (Dr Interest Expense, Cr Accrued Interest
     Payable) fails consistently. BA = 0% on all 10 problems containing this
     feature.

  3. Mixed funding (ΔFBS = -16.96): Purchases that are partially cash and
     partially financed (Dr Asset, Cr Cash, Cr Notes Payable) confuse the model's
     arithmetic. Both the Notes Payable split and the asset total are wrong.

  4. Prepaid items (ΔFBS = -23.71, present in 24/25 problems): This feature is
     nearly universal but shows the largest raw ΔFBS because it changes the sign
     of the relationship when comparing against the rare problems without it.
     Prepaid items themselves are handled reasonably (17/23 correct in ALA).

  NOTE: ΔFBS for prepaid appears large because the 1 problem without prepaid is
  L1-trivial. The directional effect should be interpreted cautiously at n=25.

PART 6 — ACCOUNT-LEVEL ACCURACY: COMPLETE BREAKDOWN
--------------------------------------------------------------------------------

Each account is evaluated across all problems where it was expected.

  -------------------------------  -----------   ---   -----   ------  -------  -------
  Cash                             asset          25    0.0%    96.0%    0.0%    47,958

  Retained Earnings                equity         25  100.0%     0.0%    0.0%         0

  Prepaid Insurance                asset          16    0.0%    37.5%    0.0%     1,167
  Furniture                        asset          14    0.0%    21.4%    0.0%     6,500
  Vehicles                         asset          14    0.0%    57.1%    0.0%    40,300
  Notes Payable                    liability      10    0.0%    90.0%    0.0%    25,289
  Accumulated Depreciation         contra_asset    9   55.6%     0.0%    0.0%         0
  Accrued Expenses                 liability       8    0.0%     0.0%    0.0%         0
  Accrued Interest Payable         liability       8    0.0%     0.0%    0.0%         0
  Allowance for Doubtful Accounts  contra_asset    4   50.0%    25.0%    0.0%       400

Top insights:

  RETAINED EARNINGS: 100% omission rate (25/25 problems).
    The model never closes revenue and expense accounts into Retained Earnings.
    This is the single most impactful failure — it causes the balance equation to
    fail in 17 of the 20 unbalanced responses. The model treats the balance sheet
    as a snapshot of raw account balances rather than a post-closing GAAP
    statement. Retained Earnings = Net Income (Revenue - Expenses) should fold
    into equity, but the model either ignores this entirely or places Revenue/
    Expense as separate line items in equity (which satisfies ALA but not BA).

  CASH: 96% arithmetic error rate, average error of $47,958.
    Cash is the most frequently transacted account and accumulates the most
    arithmetic error. The large average error reflects cascading mistakes: if
    COGS or mixed-funding transactions are computed incorrectly, the cash figure
    compounds all those errors.

  ACCUMULATED DEPRECIATION: 55.6% omission.
    Contra-asset blindness — when the model depreciation adjustments are present,
    it often records only the Depreciation Expense debit but omits the
    Accumulated Depreciation credit entry, or treats Accumulated Depreciation as
    a standard asset rather than a contra (balance-reducing) account.

  NOTES PAYABLE: 90% arithmetic error rate, average error $25,289.
    Driven almost entirely by mixed_funding transactions where the cash/loan
    split is computed incorrectly. The model either assigns the full purchase
    price to cash or gets the proportional split wrong.

  ACCRUED EXPENSES / ACCRUED INTEREST PAYABLE: 0% error rate.
    Notably, the model handles simple accrual entries perfectly when the amounts
    are given directly. This confirms the failure is conceptual (closing entries,
    contra accounts) rather than a general inability to handle liabilities.

  HALLUCINATIONS: Only 1 instance — Inventory appearing in a problem where it
    was not expected. Crucially, the hallucinated account was in-schema
    (schema_known_rate = 100%). The model does not invent fictional accounts;
    it applies real-but-wrong accounts from its training knowledge.

PART 7 — BALANCE FAILURE ROOT CAUSE ANALYSIS
--------------------------------------------------------------------------------

20 out of 25 problems (80%) have BA = 0. Root cause breakdown:

  Cause                  Count   Pct    Description
  -------------------    -----   ----   -----------------------------------------------
  omitted_key_account       17   85%    A structurally required account is missing
  missing_section            3   15%    Entire balance sheet section (liabilities) absent

  OMITTED KEY ACCOUNT (17 cases):
    All 17 are caused by missing Retained Earnings (100%).
    In 7 of 17 cases, Accumulated Depreciation is also absent.
    In 2 of 17 cases, Allowance for Doubtful Accounts is also absent.
    The imbalance ranges from -$300,000 to +$75,600 depending on net income.

  MISSING SECTION (3 cases):
    All 3 are L1 problems (FB_1_00003, FB_1_00004, FB_1_00005).
    The liabilities section is entirely absent in these responses.
    These are early-period problems where the model may treat a zero-or-small
    liability balance as negligible and omit the section entirely.
    Imbalances: $200, -$11,400, -$6,300.

  Arithmetically balanced problems (BA = 1): 5
    L1: 2 problems (FB_1_00001, FB_1_00002)
    L2: 3 problems (FB_2_00008, FB_2_00009, FB_2_00010)
    These are the simpler problems without COGS, without mixed funding, and
    without the Retained Earnings closing requirement (sometimes the model
    happens to place net income directly in Owner's Equity, which balances).

PART 9 — KEY RESEARCH FINDINGS (PAPER-READY SUMMARY)
--------------------------------------------------------------------------------

FINDING 1: SYSTEMATIC CLOSING-ENTRY BLINDNESS
  Claude Haiku 3 omits Retained Earnings in 100% of evaluated problems
  (n=25). The accounting closing cycle — debiting Revenue and crediting
  Retained Earnings, then crediting Expense and debiting Retained Earnings —
  is completely absent from the model's output. This is a CONCEPTUAL deficit,
  not arithmetic. The model knows the individual account values (Revenue,
  Expenses are often correct) but does not know that they must be collapsed into
  Retained Earnings before the balance sheet is finalised. This alone drives
  85% of all balance equation failures.

FINDING 2: CONTRA-ASSET BLINDNESS
  Accumulated Depreciation is omitted in 56% of problems where it is expected.
  Allowance for Doubtful Accounts is omitted in 50% of relevant problems.
  Both are contra-asset accounts (normal credit balance, subtracted from the
  asset section). The pattern suggests the model's account taxonomy does not
  reliably represent the contra-asset concept.

FINDING 3: TRANSACTION-TYPE SPECIFIC ERROR TRIGGERS
  Three transaction templates reliably trigger the first error:
    - cash_sale_with_cogs: 4-entry COGS transactions
    - mixed_purchase_equipment: 3-entry split-funding purchases
    - prepaid_rent: adjustment-based prepaid recognition
  These are all L3+ features. Problems using only L1-L2 templates may
  achieve BA > 0 if the closing-entry issue happens not to appear.

FINDING 4: ARITHMETIC ERRORS DOMINATE AT HIGH COMPLEXITY
  At L5, 82.5% of errors are arithmetic (correct account, wrong value).
  At L1, only 42.9% are arithmetic — omission errors are more common.
  This shift suggests the model's schema coverage improves at scale
  (it knows more accounts to include) but loses arithmetic precision as
  the number of interacting transactions grows.

FINDING 5: ERROR PROPAGATION IS GRADUAL, NOT STEP-CHANGE
  Errors do not appear suddenly at a single critical transaction; instead,
  MAE grows roughly quadratically through the problem (Q0=$0, Q25=$2K,
  Q50=$4K, Q75=$9K, Q100=$17K). This indicates the model accumulates small
  arithmetic deviations from the onset transaction forward, rather than
  catastrophically failing at one point.

FINDING 6: PARSE RELIABILITY (POSITIVE FINDING)
  With the thousands-separator comma fix, parse rate is 100% across all
  zero_shot responses. The model reliably outputs syntactically structured
  JSON, meaning evaluation failures are genuine semantic errors, not
  formatting noise. This is an important methodological control.
